# Recommendation Contract

This document defines how the backend should store and use the recommendation output generated by the ML pipeline.

## Recommendation Source

For the initial scraped dataset, recommendations are already precomputed in:

```text
recommender_system/data/precomputed_recommendations.csv
```

The backend should import this file into a database table so recommendations can be returned directly from the website API.

## Recommendation Table Schema

Recommended table name:

```text
property_recommendations
```

Required columns:

| Column | Type | Description |
| --- | --- | --- |
| source_apartment_id | integer | The apartment currently being viewed by the user. |
| recommended_apartment_id | integer | The recommended apartment. |
| rank | integer | Recommendation order. Lower rank means stronger recommendation. |
| similarity_score | float | Cosine similarity score generated by the ML pipeline. |

## Important Rules

- `source_apartment_id` must reference an existing `apartment_id` in the properties table.
- `recommended_apartment_id` must reference an existing `apartment_id` in the properties table.
- The backend should not let users create or edit recommendation rows manually.
- The backend should return recommendations ordered by `rank ASC`.
- The source apartment itself should not appear as one of its own recommendations.
- If a property is deleted, unpublished, sold, or rented, the backend should avoid returning it as a recommendation.

## Backend Query Pattern

When the user opens an apartment details page, the backend should query:

```sql
SELECT
    r.recommended_apartment_id,
    r.rank,
    r.similarity_score
FROM property_recommendations r
WHERE r.source_apartment_id = :apartment_id
ORDER BY r.rank ASC;
```

Then join `recommended_apartment_id` with the properties table to get the display fields needed by the frontend, such as:

```text
price
location
bedrooms
bathrooms
area_sqm
image
listing_type
```

## Initial Data Flow

For the current scraped data:

```text
scraped apartments
-> stored in database
-> precomputed_recommendations.csv imported into property_recommendations table
-> backend returns recommendations by source_apartment_id
```

No recomputation is needed if the scraped dataset and apartment ids stay unchanged.

## Update Flow

When new properties are added or existing properties change:

```text
latest full property inventory from database
-> ML batch script
-> refreshed recommendation rows
-> backend replaces/updates property_recommendations table
```

The update should be based on the latest full property inventory, not only newly added properties. This is important because new properties can become good recommendations for old properties too.

## Batch Command

For the current legacy scraped CSV:

```powershell
python -m recommender_system.ML.run_recommendation_batch
```

For future strict database exports:

```powershell
python -m recommender_system.ML.run_recommendation_batch --input path/to/database_export.csv --mode strict --require-identifier
```

The strict mode expects the input data to follow:

```text
recommender_system/docs/property_schema.md
```

## Output Format

The batch command writes a CSV with this format:

```text
source_apartment_id,recommended_apartment_id,rank,similarity_score
```

Example:

```csv
source_apartment_id,recommended_apartment_id,rank,similarity_score
1,597,1,0.320846
1,9,2,0.091040
2,632,1,0.940433
```

## Refresh Schedule

The recommendation system is offline. Recommendations are not recomputed on every website request.

Recommended schedule:

```text
Run the batch script once per week, or whenever a large batch of new properties is added.
```

If the database changes and the recommendation table is not refreshed, recommendations may become outdated.
